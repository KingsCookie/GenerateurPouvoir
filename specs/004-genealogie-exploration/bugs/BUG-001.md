# Bug Report: Profondeur d'arbre fiche vs page dédiée, contenu des cases, et filtre « dernière génération » par défaut

**Type**: Spec conflict + Spec gap (raffinement pré-implémentation)
**Severity**: Medium
**Feature**: 004-genealogie-exploration
**Reported**: 2026-06-10
**Status**: Patched
**Patched**: 2026-06-10

## Description

Précisions de comportement qui **affinent et corrigent** la spec actuelle (encore au stade
`spec.md`, avant `plan`/`tasks`) sur deux axes :

### A. Arbre généalogique — fiche vs page dédiée

1. **Sur la fiche individu**, la profondeur de l'arbre est **fixe = 2** (ancêtres **et**
   descendants), **non réglable**. Un bouton **« Explorer l'arbre »** ouvre une **page dédiée**.
2. **Sur la page dédiée**, l'utilisateur peut afficher **N niveaux** d'ancêtres et de descendants
   (profondeur réglable).
3. **Contenu des cases différent selon la page** :
   - **Fiche** : chaque case affiche **nom + pouvoir(s)** uniquement (pas d'âge/date).
   - **Page dédiée** : chaque case affiche **nom + âge + pouvoir(s)**.

### B. Liste de population — filtre par défaut « dernière génération » + persistance

4. La page **Liste** est **par défaut filtrée sur la dernière génération** (tranche de 20 ans la
   plus récente présente dans la population).
5. L'utilisateur peut changer pour afficher d'autres générations ; **les filtres sont conservés en
   mémoire** lors des changements de page (revenir sur la Liste restaure l'état des filtres).
6. **Défaut dynamique** : tant que l'utilisateur **n'a pas modifié manuellement** le filtre de
   génération depuis l'état initial, c'est **toujours la dernière génération** qui est affichée
   (le défaut se recale automatiquement quand le temps avance et fait apparaître une génération
   plus récente). Dès qu'un changement manuel a lieu, ce choix est respecté et persiste.

## Artifact Traceability

### spec.md

- **Affected user story** : US1 (arbre généalogique) et US2 (recherche & filtres).
- **Affected requirements** :
  - **FR-002** (CONFLIT) : « profondeur N **sélectionnable** sur la fiche, défaut 2 ». À corriger :
    sur la **fiche** la profondeur est **fixe = 2** ; la **sélection de N** se fait **uniquement
    sur la page dédiée** (+ bouton « Explorer l'arbre » depuis la fiche).
  - **FR-003** (CONFLIT) : « chaque case affiche nom, date/âge, pouvoirs » — uniforme. À scinder :
    cases **fiche** = **nom + pouvoir(s)** ; cases **page dédiée** = **nom + âge + pouvoir(s)**.
  - **FR-005** (À PRÉCISER) : la page dédiée n'affiche **pas** « la même profondeur N que la
    fiche » ; elle a sa **propre profondeur N réglable** (la fiche est figée à 2).
  - **FR-008** (GAP) : aucun **filtre par défaut** n'est défini ; ajouter « défaut = dernière
    génération ».
  - **US2 / FR-009..010** (GAP) : aucune exigence de **persistance des filtres** entre navigations
    ni de **défaut dynamique** (recalcul de la dernière génération tant qu'aucune modification
    manuelle).
- **Gap/erreur identifié** : la spec traitait la profondeur et le contenu des cases de manière
  **uniforme** fiche/page, alors que les deux diffèrent ; et le comportement par défaut + la
  persistance des filtres de la Liste n'étaient pas spécifiés.

## Root Cause Analysis

Raffinement fonctionnel post-spécification : la première spec a fait l'hypothèse raisonnable d'un
arbre **uniforme** (profondeur réglable partout, cases identiques) et n'avait pas tranché le
comportement par défaut de la Liste. L'auteur précise désormais une **distinction fiche/page
dédiée** (lecture rapide vs exploration) et un **défaut orienté usage** (voir d'abord les derniers
nés), avec **persistance** pour le confort de navigation. Aucune erreur de code (pré-implémentation).
