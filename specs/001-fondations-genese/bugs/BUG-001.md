# Bug Report: Libellé des pouvoirs de genèse non conforme au gabarit (AE/PE/PA/PR)

**Type**: Spec gap (sous-spécification du rendu) + Implementation drift (libellé incorrect)
**Severity**: Medium
**Feature**: 001-fondations-genese
**Reported**: 2026-06-09
**Status**: Patched
**Patched**: 2026-06-09

## Description

Les pouvoirs aléatoires de la **population de départ** doivent suivre **le même algorithme que la
mutation forte** (gabarit AE/PE/PA/PR) — ce qui est **déjà le cas** dans le code
([genesis.ts](../../../src/core/genesis/genesis.ts) appelle bien `generateStrongMutationPower`).

Le défaut porte sur le **rendu lisible** du pouvoir. Le libellé affiché doit respecter, **par gabarit**,
la phraséologie suivante :

| Gabarit | Composition (types) | Phrasé attendu |
|---|---|---|
| **AE** | Action + Élément | `{action} {element}` |
| **PE** | Partie du corps + État | `{partie du corps} en {etat}` |
| **PA** | Partie du corps + Ajout | `{ajout} sur {partie du corps}` |
| **PR** | Partie du corps + Remplacement | `{remplacement} à la place de {partie du corps}` |

Ces libellés doivent être **affichés proprement dans la liste des personnes** (et, par cohérence, sur la
fiche individu).

### Comportement actuel (incorrect)

- [derived.ts](../../../src/core/genesis/derived.ts) — `powerLabel()` joint les libellés de traits avec
  ` · ` : un pouvoir PE s'affiche `« Yeux · Lumineux »` au lieu de `« Yeux en Lumineux »`.
- [strongMutation.ts:47](../../../src/core/powers/strongMutation.ts#L47) — le `label` stocké vaut
  `` `${traitA.label} ${traitB.label}` `` (ordre `[typeA, typeB]`), donc :
  - **PE** → `« Yeux Lumineux »` (connecteur « en » manquant) ;
  - **PA** → `« Yeux Fourrure »` (ordre inversé + « sur » manquant ; attendu `« Fourrure sur Yeux »`) ;
  - **PR** → `« Yeux Pinces de crabe »` (ordre inversé + « à la place de » manquant ;
    attendu `« Pinces de crabe à la place de Yeux »`) ;
  - **AE** → `« contrôle feu »` ✅ (seul cas correct).

Les vues consomment `powerLabel()` (et non le `label` stocké), donc l'affichage réel est la variante
` · ` — clairement non conforme.

## Artifact Traceability

### spec.md
- **User stories affectées** : **US1** (P1) — affichage de la liste ; **US2** (P2) — fiche individu.
- **Exigences affectées** :
  - **FR-014** : définit le gabarit AE/PE/PA/PR et le tirage pondéré des traits — **correcte**, mais ne
    spécifie pas la **mise en forme** du libellé.
  - **FR-018** : « la liste … montrant … pouvoir(s) » — le format d'affichage du pouvoir n'est pas défini.
  - **FR-019** : fiche affichant « le libellé du pouvoir » — idem, format non spécifié.
- **Gap identifié** : aucune exigence ne définit **comment** rendre lisible un pouvoir par gabarit
  (connecteurs « en », « sur », « à la place de » et ordre des traits). La spec est **sous-spécifiée** ;
  l'implémentation a choisi un rendu naïf (`{a} {b}` / ` · `) incorrect pour PE/PA/PR.

  libellé** (pure, dans le cœur) doit être précisée.
- **Impact** : aucun changement d'architecture. La fonction de libellé devient **dépendante du gabarit**.

## Root Cause Analysis

**Oubli de spécification (spec oversight)** combiné à un choix d'implémentation par défaut. La source de
vérité [DescriptionProjet.md §6.1](../../../rsrc/DescriptionProjet.md) nomme les **types** composant chaque
gabarit (« Action + Élément », etc.) mais ne fournit pas, pour la mutation forte, la **chaîne de rendu**.
Les connecteurs « en / sur / à la place de » apparaissent dans l'algorithme traits→pouvoirs (§6.4) mais
n'avaient pas été remontés au niveau du libellé de genèse. Faute de règle, l'implémentation a concaténé les
traits, ce qui inverse l'ordre (PA/PR) et omet les connecteurs (PE/PA/PR).

> Cohérence avec le Principe IX : la phraséologie fournie ici **précise** (sans contredire) §6.1. Elle peut
> être intégrée aux **artefacts Spec Kit** (spec/data-model/contrat). Si l'on souhaite aussi l'inscrire dans
> `rsrc/DescriptionProjet.md`, cela requiert une **autorisation explicite** de l'auteur.
